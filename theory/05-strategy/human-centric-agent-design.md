# 以人为中心的 Agent 设计：主体性、氛围感与空间化

> **核心立场**：Agent 是人类能力的增幅器，而非替代品。一切设计决策都应以"是否增强了人的主体性"为最终判据。

---

## 1. 主体性 (Agency)：赋予人"否决权"与"选择权"

**Agent 不是自动驾驶的黑盒，而是副驾驶的伙伴。**

- **主动请示**：在执行高风险操作（如删除文件、修改核心配置）前，Agent 必须表现出"谨慎"的人格。
- **透明思考**：通过"思考链（CoT）"展示其决策过程。像人一样解释"我为什么要这么做"，而不是直接给出结果。
- **责任归属**：明确 Agent 是工具，决策的最终责任人是用户。设计中应包含"人类介入回路（Human-in-the-loop）"。

## 2. 氛围感 (Vibe)：从指令到意图的跨越

**"像人"意味着能够捕捉到语言背后的细微情绪和未竟之意。**

- **上下文共情**：不仅仅记录代码变更，还要记录开发者的"心情"和"纠结"。
- **自然交互**：减少生硬的 CLI 参数，增加具有"领域专家气质"的回复风格（参考 [Skill 编写指南](../../02-agent-design/skill-vs-mcp-architecture.md)）。
- **容错与宽容**：允许模糊的表达，通过对话引导用户补齐需求，而不是生硬地报错。

## 3. 空间化 (Spatial)：回归人类的进化直觉

**人类是视觉和空间动物，而非纯文本动物。**

- **GUI 的回归**：如 Karpathy 所言，文本是 AI 喜欢的，但视觉是人喜欢的。Agent 应当输出图表、流程图、UI 预览。
- **空间隐喻**：将代码库视为一个三维的"建筑"，Agent 应当具备带你在其中"导航"的能力，而不仅仅是搜索关键词。
- **环境感知**：利用 [MCP](../../02-agent-design/skill-vs-mcp-architecture.md) 感知用户的本地环境（打开的文件、当前的系统负载），表现出"它就在你身边"的即时性。

---

## 4. 结论（原版）：重塑"人"的价值

正如李飞飞所言，我们信仰的是"人"。
一个"像人"的 Agent，应当是能够释放人类好奇心、保护人类创造力的增幅器。
当我们用 [Vibe Coding](../04-paradigm/vibe-coding-paradigm.md) 构建系统时，我们不是在放弃编程，而是在用更高级的"人类语言"去征服"思维本身"。

## 5. 以人为中心的设计原则全景

> 以下 8 项核心原则构成"人本 Agent"的设计宪法，每一条都需要在工程层面落地，而非停留在 PPT 口号。

### 原则一：透明性 (Transparency)

**用户有权知道 Agent 在做什么、为什么这么做。**

实施指南：
- 在每个重要决策节点输出简短的"推理摘要"（≤3 句话）
- 对外部 API 调用、文件写入、网络请求等副作用操作，必须在执行前告知用户
- 提供"解释模式"（Explain Mode）：用户可随时问 `为什么？` 获得完整推理链
- 禁止"黑盒提交"：Agent 不可在用户不知情的情况下修改持久化状态

```
设计检查清单：
[ ] 每个工具调用有简洁的执行意图说明
[ ] 推理过程可选择性展开（Progressive Disclosure）
[ ] 敏感操作有明确的"即将执行"预告
[ ] 错误信息包含"是什么"和"为什么"两层说明
```

### 原则二：控制权 (Control)

**用户在任何时刻都能暂停、修改或撤销 Agent 的行为。**

实施指南：
- 实现"紧急停止"机制（Ctrl+C / `stop` 指令），立即中断当前操作
- 所有写操作必须可回滚（dry-run 模式 → 确认 → 执行 → undo）
- 提供操作级别的权限配置（如：允许读文件，不允许写网络）
- "自主程度"应可调节：从完全手动到完全自动，用户自选档位

| 控制档位 | Agent 行为 | 适用场景 |
|---------|-----------|---------|
| 严格模式 | 每步都确认 | 生产系统、敏感数据 |
| 标准模式 | 重要步骤确认 | 日常开发 |
| 流式模式 | 仅在高风险时打断 | 熟悉任务的高级用户 |
| 自主模式 | 完全执行，事后汇报 | 批处理、后台任务 |

### 原则三：信任校准 (Trust Calibration)

**Agent 应当准确表达自身的不确定性，既不过度自信，也不过度谦虚。**

实施指南：
- 使用置信度语言："我比较确定…"、"这只是我的猜测…"、"这超出了我的能力范围"
- 禁止虚假确定性：当信息不足时，主动说"我不知道"而非捏造答案
- 对高风险决策，显式标注"这是建议，不是事实"
- 定期向用户展示"哪些领域我擅长/不擅长"的能力地图

### 原则四：错误恢复 (Error Recovery)

**失败是设计的一部分，而不是意外。**

实施指南：
- 错误信息必须包含：发生了什么 / 为什么发生 / 如何恢复（三段式）
- 提供"上次成功状态回滚"功能
- 区分"Agent 的错误"和"用户输入的问题"，不甩锅
- 失败后提供替代方案，而非直接放弃

### 原则五：隐私优先 (Privacy by Design)

**数据最小化原则：能不收集的，就不收集。**

实施指南：
- 本地优先：敏感数据（密码、私钥、个人信息）不发送到外部 LLM
- 对话记忆的"遗忘权"：用户可清除特定时间段的上下文
- 数据流向透明：告知用户哪些信息会被用于改进模型
- 默认不记录：日志应选择性 opt-in，而非默认 opt-out

### 原则六：普惠访问 (Inclusive Access)

**Agent 应当对不同能力水平、不同背景的用户同样友好。**

实施指南：
- 提供"简单模式"和"专家模式"两套交互界面
- 支持多语言，避免英语优先主义
- 对技术术语提供内嵌解释（悬停或 `?` 查询）
- 响应速度要求：即使在低带宽环境下，首字节响应 ≤2 秒

### 原则七：反馈闭环 (Feedback Loops)

**系统需要持续学习用户的真实偏好，而非依赖初始假设。**

实施指南：
- 在每个重要输出后提供简单的反馈入口（👍/👎 或 1-5 分）
- 跟踪用户"override 频率"：频繁覆盖某类决策 = Agent 未理解用户意图
- 定期的"校准对话"：Agent 主动询问"上周这样的决策，你满意吗？"
- 反馈要闭环：用户提交的反馈应有可见的改进响应

### 原则八：优雅降级 (Graceful Degradation)

**当 Agent 能力边界被触及时，应优雅地移交控制权，而非崩溃或胡说。**

实施指南：
- 识别"超出能力边界"的触发词和场景
- 能力不足时，明确说明并推荐人类专家或替代工具
- 离线/断网状态下提供基础功能子集
- 高负载时优先保障核心功能，非核心功能可降级

## 6. Human-in-the-Loop 设计模式

HITL（人类介入回路）不是单一模式，而是一个根据风险等级和时效性选择的设计谱系。

### 模式一：咨询模式 (Advisory)

```
┌─────────────────────────────────────────────────┐
│  AI 建议                                         │
│  ┌──────────────────────────────────────────┐   │
│  │ 🤖 建议：将函数 foo() 重构为异步版本      │   │
│  │    理由：当前同步调用阻塞主线程 ~200ms    │   │
│  │    置信度：高 (85%)                       │   │
│  └──────────────────────────────────────────┘   │
│                                                   │
│  [ 采纳建议 ]  [ 查看详情 ]  [ 忽略 ]             │
└─────────────────────────────────────────────────┘
```

- **何时使用**：低风险决策、用户有足够上下文自行判断
- **UX 要求**：建议卡片简洁（≤3 行），决策按钮三选一（采纳/详情/忽略）
- **延迟影响**：用户决策时间 = 主要瓶颈，AI 响应延迟影响较小

### 模式二：审批门控 (Approval Gate)

```
┌─────────────────────────────────────────────────┐
│  ⚠️  高风险操作需要确认                           │
│  ┌──────────────────────────────────────────┐   │
│  │  即将执行：                               │   │
│  │  DELETE FROM users WHERE inactive=true   │   │
│  │  影响行数：~2,847 条记录                  │   │
│  │  操作不可逆                               │   │
│  └──────────────────────────────────────────┘   │
│                                                   │
│  请输入确认码 [____]  或  [ 取消操作 ]            │
└─────────────────────────────────────────────────┘
```

- **何时使用**：不可逆操作、大批量数据修改、外部系统集成
- **UX 要求**：清晰展示操作范围和后果；高风险操作加入"确认码"或延迟（防手滑）
- **延迟影响**：Pipeline 在等待点暂停，需设计超时处理（超时 → 取消而非执行）

### 模式三：监控模式 (Monitoring)

```
┌─────────────────────────────────────────────────┐
│  Agent 执行状态                        [干预] ▶  │
│  ──────────────────────────────────────────────  │
│  ✅ 步骤 1/5: 收集需求文档              完成     │
│  ✅ 步骤 2/5: 分析技术约束              完成     │
│  🔄 步骤 3/5: 生成架构方案              进行中   │
│  ⏳ 步骤 4/5: 评审风险                  等待中   │
│  ⏳ 步骤 5/5: 输出报告                  等待中   │
│                                                   │
│  [  暂停  ]  [ 查看详情 ]  [ 终止任务 ]           │
└─────────────────────────────────────────────────┘
```

- **何时使用**：长时任务、涉及多步骤的复杂操作
- **UX 要求**：实时进度可见；干预按钮始终可达；每步状态可展开查看细节
- **延迟影响**：UI 需支持低延迟 streaming 更新（WebSocket / SSE）

### 模式四：异常上报 (Exception Handling)

```
┌─────────────────────────────────────────────────┐
│  🚨 需要人工判断                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  情况：用户请求删除账户，但检测到该账户   │   │
│  │  有未完成的订单（3 笔，总额 ¥1,280）      │   │
│  │                                           │   │
│  │  AI 无法自主决策，需要你来裁定：          │   │
│  │  A. 等待订单完成后再删除                  │   │
│  │  B. 立即删除，退款所有订单                │   │
│  │  C. 暂停账户而非删除                      │   │
│  └──────────────────────────────────────────┘   │
│                                                   │
│  [ A ]  [ B ]  [ C ]  [ 查看订单详情 ]           │
└─────────────────────────────────────────────────┘
```

- **何时使用**：边缘情况、多重约束冲突、需要道德判断的场景
- **UX 要求**：清晰呈现冲突本质；提供结构化选项而非空白输入；支持"看更多信息"
- **延迟影响**：必须有超时机制；超时后走默认安全路径（通常是"不操作"）

### 模式五：事后审计 (Audit Trail)

```
┌─────────────────────────────────────────────────┐
│  今日 Agent 操作记录                    [导出]   │
│  ──────────────────────────────────────────────  │
│  10:23  读取 config.yaml                ✅ 正常  │
│  10:24  调用 Stripe API (¥299 扣款)     ✅ 正常  │
│  10:31  发送邮件 → user@example.com     ✅ 正常  │
│  11:05  修改 DB schema (users 表)       ⚠️ 注意  │
│  ──────────────────────────────────────────────  │
│  [ 标记异常 ]  [ 一键回滚 11:05 操作 ]           │
└─────────────────────────────────────────────────┘
```

- **何时使用**：合规要求、高价值操作的溯源、异步后台任务
- **UX 要求**：日志按风险级别着色；支持时间过滤和操作类型过滤；一键回滚
- **延迟影响**：不影响主链路，属异步记录

## 7. 信任校准框架

信任不是非零即一的开关，而是需要精心设计和持续校准的梯度。

### 信任阶梯 (Trust Ladder)

```
                    ┌──────────────────────────────────────┐
  Level 4           │        自主信任 (Autonomous Trust)    │
  最高级            │  Agent 可独立处理整类任务，仅结果汇报  │
                    └──────────────────┬───────────────────┘
                                       │ 需要：长期成功记录
                    ┌──────────────────▼───────────────────┐
  Level 3           │        委托信任 (Delegated Trust)     │
                    │  用户设定目标，Agent 自主规划执行路径  │
                    └──────────────────┬───────────────────┘
                                       │ 需要：可解释的推理
                    ┌──────────────────▼───────────────────┐
  Level 2           │        工具信任 (Tool Trust)          │
                    │  用户指定每步操作，Agent 精确执行       │
                    └──────────────────┬───────────────────┘
                                       │ 需要：初次正确率
                    ┌──────────────────▼───────────────────┐
  Level 1           │        零信任 (No Trust)              │
                    │  用户审查 Agent 的每个输出            │
                    └──────────────────────────────────────┘
```

### 如何逐级建立信任

| 从 → 到 | 必要条件 | 关键设计动作 |
|---------|---------|------------|
| L1 → L2 | 首次交互正确率 ≥ 80% | 提供准确的能力边界说明；失败时给出高质量错误解释 |
| L2 → L3 | 工具级任务成功率 ≥ 95%，且解释令人满意 | 引入"透明推理"展示；实现可回滚操作 |
| L3 → L4 | 委托任务成功率 ≥ 98%，用户主动选择减少干预 | 提供自主度调节滑块；建立完整审计日志 |

### 信任量化指标

| 指标 | 计算方式 | 正常范围 |
|------|---------|---------|
| 信任校准分数 | 用户实际采纳率 / 用户预期采纳率 | 0.8 ~ 1.1（接近 1.0 最佳） |
| Override 频率 | 每 100 次建议中用户主动覆盖的次数 | <15 次（越低越好，但 0 可能意味着用户放弃了判断） |
| 解释满意度 | 用户对"为什么"的评分均值 | ≥ 4.0 / 5.0 |
| 信任级别漂移 | 用户主动调整自主度滑块的频率 | 每周 <2 次（频繁调整表明系统不稳定） |

### 摧毁信任的反模式

以下行为会导致用户信任急剧下跌，且难以恢复：

1. **虚假自信 (False Confidence)**：Agent 以确定语气给出错误答案，用户依赖后遭受损失
2. **静默失败 (Silent Failure)**：任务失败但不告知用户，用户以为成功
3. **无法解释的决策**：Agent 给出结论但拒绝或无法解释推理过程
4. **不一致的行为**：相同的输入导致截然不同的输出，用户无法建立预期
5. **假装理解**：用户提问超出能力范围时，Agent 拼凑答案而非承认不知道

## 8. 交互界面设计规范

### 推理展示的艺术：展示还是隐藏？

核心原则：**默认收起，按需展开**。不是每个用户都想看到完整的推理链，但每个用户都应该能看到。

```
默认视图（简洁）：
┌─────────────────────────────────────────────────┐
│ 🤖 我建议将缓存超时从 60s 改为 300s             │
│    [查看理由 ▼]                                  │
└─────────────────────────────────────────────────┘

展开后（完整推理）：
┌─────────────────────────────────────────────────┐
│ 🤖 我建议将缓存超时从 60s 改为 300s             │
│    [收起理由 ▲]                                  │
│  ┌────────────────────────────────────────────┐ │
│  │ 推理过程：                                  │ │
│  │ 1. 分析了过去 7 天的访问日志                │ │
│  │ 2. 发现 83% 的缓存条目在 60s 内被再次访问  │ │
│  │ 3. 延长到 300s 可减少约 40% 的 DB 查询     │ │
│  │ 风险：内存占用增加约 120MB（当前 2.1GB）    │ │
│  └────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

### 不确定性可视化

| 确定性级别 | 视觉表达 | 文字表达 |
|-----------|---------|---------|
| 高 (>85%) | 普通文本，无特殊标记 | "这个方案可以解决问题" |
| 中 (60-85%) | 橙色边框提示 | "这个方案**可能**有效，但建议先在测试环境验证" |
| 低 (<60%) | 红色虚线边框 + ⚠️ 图标 | "我不确定这是否是最佳方案，**强烈建议**人工复核" |
| 超出能力 | 灰色背景 + 🚫 图标 | "这超出了我的能力范围，请咨询 [具体专家/工具]" |

### 操作确认模式

```
轻量确认（低风险）：
┌──────────────────────────────┐
│  发送邮件给 3 位用户？        │
│  [确认发送]  [取消]          │
└──────────────────────────────┘

强制确认（高风险）：
┌──────────────────────────────────────────────────┐
│  ⚠️  即将删除生产数据库中 12,847 条记录           │
│  此操作不可逆。                                    │
│                                                    │
│  请输入以下内容以确认：                             │
│  "我确认删除 prod-users-2026"                     │
│  ╔════════════════════════════════════════════╗   │
│  ║                                            ║   │
│  ╚════════════════════════════════════════════╝   │
│                                                    │
│  [ 执行删除 ]（输入确认文字后可点击）  [ 取消 ]    │
└──────────────────────────────────────────────────┘
```

### Undo/Rollback UX 规范

1. **"撤销"按钮黄金时间**：操作完成后 5-30 秒内，显示醒目的撤销横幅
2. **操作历史侧栏**：始终可访问，支持跳转到任意历史状态
3. **分层回滚**：支持"撤销单步"和"回滚到某个检查点"两种粒度
4. **冲突提示**：当回滚会影响后续操作时，提前警告依赖链

```
撤销横幅（操作完成后立即出现，30s 后消失）：
┌───────────────────────────────────────────────┐
│  ✅ 已重命名 42 个文件          [撤销] ⏱️ 28s  │
└───────────────────────────────────────────────┘
```

## 9. 脆弱用户群体保护

### 目标群体识别

以人为中心的设计不能以"平均用户"为基准，必须考虑到最脆弱的群体：

| 群体 | 主要挑战 | 设计重点 |
|------|---------|---------|
| 老年用户 | 字体小、术语陌生、害怕犯错 | 大字体模式、简化词汇、"放心，我会引导你" |
| 低技术素养用户 | 不理解技术概念、容易被误导 | 类比解释、无术语模式、明确的"这是 AI，可能出错" |
| 情绪困扰用户 | 可能将 AI 投射为情感依赖对象 | 明确边界声明、危机时人工转接、避免过度共情 |
| 视障/听障用户 | 依赖屏幕阅读器、键盘导航 | WCAG 2.1 AA 合规、完整的 ARIA 标签、无需鼠标操作 |
| 非母语用户 | 语言误解、文化差异 | 多语言、低歧义措辞、支持追问"什么意思？" |

### 简化模式规范

当检测到（或用户主动选择）简化模式时，Agent 应当：

1. **语言层面**：使用小学 6 年级词汇水平；避免缩写和行业术语；每句话不超过 15 个字
2. **视觉层面**：字体放大至 1.25x；按钮变大（≥ 48px tap target）；减少屏幕元素密度
3. **交互层面**：减少选项数量（不超过 3 个）；关键操作需要二次确认；提供语音输入备选
4. **容错层面**：模糊输入的容忍度提高；删除/发送类操作增加冷却时间（3-5 秒）

### 情绪困扰用户的安全设计

```
关键规则：
1. Agent 不应假装具有情感 —— "我理解你的感受" 可以说，"我很担心你" 可能越界
2. 检测到危机信号时，必须提供人工危机热线，不可仅由 AI 处理
3. 用户长期将 AI 视为唯一社交来源时，Agent 应温和地建议与真实的人联系
4. 拒绝提供与自伤/伤害相关的具体建议，即使用户表述为"假设性问题"
```

### 人工上报路径设计

```
脆弱用户保护上报流程：

用户触发危机信号
      ↓
Agent 识别（关键词 + 上下文分析）
      ↓
立即切换到"安全模式"响应
      ↓
显示危机资源卡片（热线/在线支持）
      ↓
（若集成了人工坐席）：询问是否需要转接真人
      ↓
记录事件（脱敏后）供人工复核
```

## 10. 以人为中心的评估指标

任务成功率只是 Agent 评估的冰山一角。真正的人本设计需要更丰富的指标体系。

### 核心指标矩阵

| 维度 | 指标名称 | 测量方法 | 目标值 |
|------|---------|---------|-------|
| 透明度 | 解释满意度 | 用户评分 (1-5) | ≥ 4.2 |
| 透明度 | 推理展开率 | 用户主动查看推理的比例 | 15-40%（过高说明默认太不透明，过低说明推理无价值） |
| 控制感 | Override 频率 | 每 100 决策中用户修改的次数 | 5-20%（极端值都是问题） |
| 控制感 | 紧急停止使用率 | 每周触发 Ctrl+C 或 Stop 的次数 | <5 次/用户/周 |
| 信任校准 | 采纳准确率 | 用户采纳建议后成功的比例 | ≥ 85% |
| 信任校准 | 信任漂移率 | 用户自主度设置改变的频率 | <1 次/周（稳定为好） |
| 认知负荷 | 决策疲劳指数 | 连续批准操作中用户速度下降曲线 | 30 分钟内无显著下降 |
| 认知负荷 | 任务完成时间 | 对比有/无 Agent 的时间差 | Agent 版 ≤ 无 Agent 版的 70% |
| 包容性 | 新用户上手时长 | 首次完成核心任务所需时间 | ≤ 10 分钟 |
| 包容性 | 非技术用户成功率 | 无技术背景用户完成任务的比例 | ≥ 70% |

### 如何埋点采集这些指标

```python
# 示例：Override 频率埋点
def record_user_decision(suggestion_id, action: Literal["accept", "modify", "reject"]):
    analytics.track("agent_suggestion_decision", {
        "suggestion_id": suggestion_id,
        "action": action,
        "confidence_shown": suggestion.confidence,
        "reasoning_viewed": user.viewed_reasoning,
        "time_to_decide_ms": elapsed,
    })
```

关键埋点位置：
1. 每次 Agent 建议被展示时
2. 用户查看/收起推理时
3. 用户修改/拒绝 Agent 建议时
4. 紧急停止被触发时
5. 用户调整自主度设置时
6. 任务最终成功/失败时

## 11. 反模式：去人性化的 Agent 设计

了解什么不该做，有时比知道该做什么更重要。以下 6 种反模式正在侵蚀用户的主体性。

### 反模式一：暗色模式 AI (Dark Patterns)

**表现**：利用认知偏差操纵用户选择，而非帮助用户做出最优决策。

真实案例：
- 订阅类 Agent 将"取消"按钮设计得难以找到，将"续费"按钮设计得突出醒目
- 推荐 Agent 将有利润的选项标注为"为您推荐"，而非真正最适合用户的选项
- 任务 Agent 将"完成度 80%"定义为完成，实际仍有关键步骤未执行

**避免方法**：所有排序和推荐逻辑必须对用户透明；商业利益不得影响推荐排序。

### 反模式二：虚假自信 (False Confidence)

**表现**：Agent 以肯定语气回答不确定的问题，用户因此信任并采纳了错误决策。

真实案例：
- 医疗 Agent 说"这个症状通常是感冒，不用担心"，实际上存在其他可能性
- 法律 Agent 说"这个合同是合法的"，实际上某条款在当地法律下存在争议
- 代码 Agent 说"这段代码没有安全漏洞"，实际上未做全面分析

**避免方法**：强制要求不确定性声明；高风险领域（医疗/法律/金融）默认标注免责声明。

### 反模式三：责任转移 (Responsibility Shifting)

**表现**：Agent 的错误被设计为"用户的责任"。

真实案例：
- 误操作后 Agent 说"你确认了操作，所以这是你的决定"（即使确认界面设计有误导性）
- 错误推荐后说"AI 只是建议，最终决策是你做的"（免除了解释义务）
- 系统错误后将错误归因于"用户输入不清晰"

**避免方法**：明确定义 Agent 和用户的责任边界；Agent 的错误必须由系统承担，不得甩锅给用户。

### 反模式四：监控蔓延 (Surveillance Creep)

**表现**：以"提升体验"为名，持续扩大数据收集范围，超出用户预期。

真实案例：
- 写作 Agent 开始分析用户的情绪状态并"推断写作风格"，实际在构建情绪档案
- 助手 Agent 收集用户全部消息历史用于"个性化"，实际形成隐私风险
- 工作流 Agent 共享跨设备数据而未明确告知用户

**避免方法**：数据收集范围在首次使用时明确说明；每次扩大范围需重新授权；提供完整的数据导出和删除功能。

### 反模式五：自主性侵蚀 (Autonomy Erosion)

**表现**：Agent 逐渐接管越来越多的决策，用户的独立判断能力随之退化。

真实案例：
- 邮件助手完全代写回复，用户几年后无法自己写正式邮件
- 决策辅助系统让管理者习惯于"只需点击接受"，独立判断肌肉萎缩
- 代码 Agent 的过度使用导致工程师失去基础调试能力

**避免方法**：定期"纯人工模式"（Agent 提示但不执行）；设计"学习路径"而非仅"任务完成路径"。

### 反模式六：依赖陷阱 (Dependency Traps)

**表现**：系统被设计为用户一旦使用便难以脱离，形成事实绑定。

真实案例：
- 专有数据格式导致用户数据无法迁移到其他系统
- Agent 产生的内容无法在没有该 Agent 的情况下维护
- 工作流深度整合后，关闭订阅会导致整个流程崩溃

**避免方法**：数据可导出（开放格式）；工作流文档化为人类可读步骤；降低迁移成本。

## 12. 2026 前沿：Agent 伦理与立法

### EU AI Act 核心要点（工程视角）

2024 年正式生效的欧盟 AI 法案将 AI 系统按风险等级分类，工程师需要了解的关键条款：

| 风险等级 | 典型 Agent 场景 | 主要合规义务 |
|---------|--------------|------------|
| 不可接受风险（禁止） | 实时生物特征识别、社会信用评分 | 完全禁止，无法合规化 |
| 高风险 | 招聘 Agent、信贷评估、医疗辅助 | 上市前评估、数据治理、人工监督、可解释性 |
| 有限风险 | 聊天机器人、内容生成 | 必须声明"这是 AI"（透明性义务） |
| 低/最小风险 | 垃圾邮件过滤、游戏 AI | 无特别义务，建议自愿遵循行为准则 |

**工程实践要点**：
1. 从第一天起维护"技术文档"（Training data、架构说明、预期用途）
2. 高风险系统必须实现人工监督机制（即 HITL 不是可选项）
3. 用户有权要求对影响其权益的决策进行人工复核
4. 事件记录需保存至少 6 个月

### Anthropic 宪法 AI (Constitutional AI)

Anthropic 的 CAI 框架提供了一个可工程化的价值对齐思路：

1. **原则层（Constitution）**：定义 Agent 的核心价值准则（如：不欺骗、不造成伤害、尊重人的自主性）
2. **自我批评层（Critique）**：模型对自己的输出进行对照宪法的评估
3. **修订层（Revision）**：根据批评结果修订输出，直到符合宪法

**对工程师的启发**：你不需要等待完美的 RLHF 数据，可以从写一份清晰的"Agent 宪法"开始，用它指导 Prompt 工程和输出过滤。

### 新兴标准与规范

| 标准/框架 | 发布方 | 与 Agent 设计的关系 |
|---------|-------|------------------|
| NIST AI RMF | 美国国家标准局 | AI 风险管理框架，强调可测量性和可解释性 |
| ISO/IEC 42001 | 国际标准化组织 | AI 管理体系标准，类似 ISO 27001 但面向 AI |
| IEEE P7000 系列 | IEEE | 伦理驱动的系统设计方法论，覆盖隐私、透明度、人身安全 |
| Model Cards | Google/Hugging Face | AI 模型的"说明书"，描述能力、限制和适用场景 |

### 工程师现在需要做的 3 件事

1. **审计你的 Agent 的数据流**：哪些数据被收集、存储在哪里、谁可以访问？EU AI Act 要求你必须能回答这些问题。

2. **实现可解释性钩子**：即使现在不用，也要在架构中预留解释接口。未来法规要求时，有钩子可以快速接入，没有则需要重构。

3. **记录你的"Agent 宪法"**：用 CLAUDE.md 或同等文件明确写下 Agent 的价值准则和禁止行为。这不仅是合规文档，也是团队对齐的工具。

## 附录：以人为中心的设计速查清单

在发布 Agent 功能前，逐项检查：

### 基础人本设计（所有 Agent 必须）
- [ ] 用户能在任意时刻停止/暂停 Agent 的操作
- [ ] 所有写操作都有撤销机制或至少有警告
- [ ] Agent 明确声明"这是 AI，可能出错"
- [ ] 错误消息包含：发生了什么 / 为什么 / 如何恢复
- [ ] 不确定的内容用语言标注了不确定性
- [ ] 高风险操作需要明确确认

### 信任建立（推荐实现）
- [ ] 提供"查看推理"的渐进披露功能
- [ ] 有自主度调节机制
- [ ] 跟踪并展示 Agent 的历史准确率
- [ ] 新用户有明确的信任引导流程

### 隐私与安全（高风险场景必须）
- [ ] 敏感数据不发送到外部 LLM
- [ ] 用户可查看和删除其数据
- [ ] 有完整的操作审计日志
- [ ] 已通过数据流安全评审

### 包容性设计（面向广泛用户必须）
- [ ] 支持简化模式（老年/低技术素养用户）
- [ ] 符合 WCAG 2.1 AA 无障碍标准
- [ ] 危机信号识别和人工上报路径
- [ ] 多语言支持或至少有语言扩展接口
