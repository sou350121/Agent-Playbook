# Vibe Coding 范式：从意图驱动开发到 Agentic Engineering

> 📌 04-paradigm — 范式转变

**X-Ray 洞见**：Vibe Coding 不是终点，它是人类从"告诉计算机怎么做"跨越到"告诉计算机要什么"这一历史性转变的**门槛时刻**。真正的范式转变在门槛之后：当 vibe 变成 engineer 的日常，工程化约束才是下一个战场。

---

## 1. Software 1.0 → 2.0 → 3.0：完整演化图

Karpathy 在 2025 年 YC AI Startup School 演讲中系统提出三代范式，这是理解 Vibe Coding 的基础框架：

```
┌─────────────────────────────────────────────────────────────────────┐
│  Software 1.0   │  Software 2.0      │  Software 3.0               │
├─────────────────────────────────────────────────────────────────────┤
│  显式规则        │  神经网络权重       │  自然语言意图                │
│  人写每一行逻辑  │  训练数据决定行为   │  LLM 决定实现路径            │
│  确定性执行      │  统计推断          │  概率性生成                  │
│  逻辑即代码      │  逻辑即权重        │  逻辑即 prompt               │
├─────────────────────────────────────────────────────────────────────┤
│  "告诉计算机怎么做" ──────────────► "告诉计算机要什么"             │
└─────────────────────────────────────────────────────────────────────┘
```

**关键转变**：
- 1.0 → 2.0：程序员不再写规则，改为收集数据；程序行为由数据集决定，不再由人直接编写
- 2.0 → 3.0：不再需要数据集标注，改为写自然语言 prompt；LLM 成为新的"操作系统"——Software 3.0 is eating 1.0 and 2.0

> Karpathy："这是 70 年来软件范式首次根本性变化。"

## 3. Vibe Coding 的局限性：为什么它不够

### 核心问题

| 维度 | Vibe Coding | Agentic Engineering |
|------|-------------|---------------------|
| **Scope 定义** | 无边界约束，意图模糊即发散 → T1 Scope Leak | 明确 task boundary，scope 锁定再执行 |
| **Eval Loop** | 无验证机制，"能跑"≠"对" → 语义回归无法检测 | 自动测试 + eval loop，回归可量化 |
| **Trust Tier** | 无权限分层，Agent 可意外写入生产 → T4 Authority 越界 | 工具权限分级，写操作需审批 |
| **Context 管理** | 单轮/短窗口，长任务上下文断裂 | 跨轮 context 工程，状态持久化 |
| **可维护性** | 40%+ 初级开发者不理解自己部署的代码 | 人类必须能 review、解释并拥有代码 |
| **安全** | AI 生成代码安全缺陷率 24.7%，逻辑错误率 2.74× | 代码扫描 + trust boundary 强制执行 |
| **幂等性** | 无重试保障，副作用不可控 | idempotent ops + checkpoint 设计 |
| **失败恢复** | 失败即重跑，无 DAG 感知 | failure taxonomy (T1-T6) + 自愈路径 |

### 失败案例解剖

- **Scope Leak (T1)**：无 scope 定义的 vibe 会让 Agent 不断扩展功能边界，最终生成超出需求的庞大代码库——而开发者无力 review
- **语义回归**：无 eval loop 时，提示词的微小变化可能导致行为静默退化，只有上线后才被用户发现
- **意外生产变更**：无 trust tier 时，Agent 持有 full credentials，"帮你优化"可能变成"帮你清空"

## 5. 范式转变的不可逆信号

判断一个范式是否真正转变，看三个指标：

### 信号 1：工具链替换基础设施
当 Cursor/Windsurf 等 agentic IDE 的用户数超越传统 IDE 在同等细分市场的历史峰值，说明工具层已发生结构性迁移。不是"新工具加速旧工作流"，而是"旧工作流被消解"。

### 信号 2：经济逻辑反转
Vibe Coding 第一定律（来自 @nake13）：**让 AI 永远优先考虑最好的方案，哪怕它看起来最费时**——因为 AI 不疲劳，"费时"只消耗 Token。当"人力成本 vs Token 成本"的比较逻辑被广泛接受并驱动决策，范式已在经济层锚定。

### 信号 3：新问题取代旧问题
旧问题："我怎么写这段代码？"
新问题："我怎么给 Agent 定义 scope？怎么设计 eval loop？怎么分配 trust tier？"
当开发者社区的主流讨论从前者转向后者，范式转变已经完成——新问题是旧框架无法回答的，这是不可逆的标志。

## 6. Vibe Coding：意图驱动开发的终极形态

**Vibe Coding** 是指开发者通过描述"氛围"（意图、目标、约束），让 Agent 自主构建、运行并调试程序的开发模式。

- **代码的短暂性 (Ephemeral Code)**：代码不再是需要小心维护的"资产"，而是随时可以重写、生成的"耗材"。
- **角色转型**：开发者从"打字员"转向"架构师"与"审美决策者"。

## 7. 为什么本地 Agent（如 Claude Code）赢了？

Karpathy 指出，OpenAI 在 Agent 策略上可能判断失误：他们过度关注云端容器，而忽视了本地环境。

- **本地优势**：Agent "住在你电脑里"，拥有完整的本地上下文、执行权和低延迟反馈，这比云端沙盒更像一个真实的"伙伴"。
- **交互极简**：Claude Code 这种极简的 CLI 形态，反而比复杂的 Web 界面更能体现 Agent 的本质。

## 8. LLM 的 GUI：Nano Banana 范式

聊天框（CLI）只是过渡形态，人类最终偏好的是空间化、视觉化的信息。

- **超越 Markdown**：LLM 应该直接产出图表、白板、网页应用甚至动画。
- **视觉化计算**：模型不只是输出文本，而是将世界知识、逻辑与视觉表达深度融合。

## 9. 信任交付：清理你的"经验包袱"

六叔 (@nake13) 提到一个深刻的观点：Vibe Coding 是年轻人的巨大机会，因为他们没有"经验包袱"。

- **丢掉手动控权的执念**：老牌程序员习惯掌控每一行逻辑，这种"控制欲"在 Agent 时代会变成效率的绊脚石。
- **白纸优势**：没有旧习惯的人能直接把信任交付给 Agent。信任不是盲从，而是接受"随机性"并用"可验证机制"去对冲它。
- **门槛的下移**：技术壁垒正在崩塌，**"学习意愿"与"自驱力"**成了唯一的入场券。

## 10. 实战启示（2026 版）

- **随手 Vibe，但先定 Scope**：养成"为了定位一个 Bug，随手 Vibe 一个临时应用"的习惯——但先用一句话锁定边界，避免 Scope Leak。
- **从"白纸"视角思考**：即使你有十年经验，在面对新任务时，试着先像个新手一样，把整个意图抛给 Agent，看它如何构建，而不是一开始就规定它必须用什么库。
- **拥抱本地工具**：优先使用能直接读写本地文件系统的 Agent。
- **关注视觉呈现**：在提示词中要求 Agent 产出具备"空间感"的结果。
- **加入 Eval Loop**：每个 vibe session 后问自己：这段代码我能解释吗？如果不能，建立自动测试先。
- **设计 Trust Tier**：给 Agent 最小必要权限；生产写操作必须经人工确认。

---

> 💡 **总结**：Software 3.0 把"意图"变成了编程语言，Vibe Coding 是第一批母语使用者——但母语使用者也需要语法和工程化。从 Vibe 到 Agentic Engineering，不是否定，而是成熟。代码变耗材，流程和意图才是核心；Agent 变团队，scope 和 trust 才是护城河。
